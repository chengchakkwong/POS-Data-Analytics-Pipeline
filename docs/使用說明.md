# POS 數據分析系統 — 使用說明書

本文件為**內部操作手冊**，供實際使用本系統的人員依步驟執行。若需專案技術說明與架構，請參閱專案根目錄的 `README.md`。

---

## 一、前置準備

### 1. 安裝環境

- 本機需已安裝 **Conda**。
- 在專案根目錄（與 `pos_system_v2.py` 同一層）開啟終端機，執行：

  ```bash
  conda env create -f environment.yml
  conda activate <env-name>
  ```

- `<env-name>` 請依 `environment.yml` 第一行的 `name:` 為準（例如 `washing_da`）。

### 2. 設定資料庫連線

- 在專案根目錄建立 **`.env`** 檔案（若尚未建立）。
- 填入 POS SQL Server 的連線資訊，格式如下：

  ```
  DB_DRIVER={ODBC Driver 17 for SQL Server}
  DB_SERVER=你的伺服器位址
  DB_DATABASE=你的資料庫名稱
  DB_UID=你的帳號
  DB_PWD=你的密碼
  DB_TRUST_CERT=yes
  ```

- 請確認可從本機連線至 POS 資料庫（防火牆、VPN、權限等）。

### 3. 第一次執行前檢查

- 專案根目錄下會有 `data/` 資料夾；若無，執行主程式後會自動建立。
- 不需手動建立 `data/processed/` 或 `data/insights/`，程式會自動建立。

## 二、日常使用：資料同步（必做）

**目的**：從 POS 抓取「今日庫存」與「銷售歷史」，寫入本機快取，供後續分析與預測使用。

**指令**：

```bash
python pos_system_v2.py
```

**執行後會發生的事**：

- 連線至 POS 資料庫。
- 產生資料字典 → `data/processed/table_structure.txt`。
- 抓取商品庫存主檔 → 寫入 **`data/processed/DetailGoodsStockToday.csv`**。
- 增量同步銷售紀錄 → 寫入 **`data/processed/vw_GoodsDailySales_cache.parquet`**（第一次可能為全量同步，較久）。

**建議**：排程或每日固定執行一次，確保本地資料為最新，再跑底下的分析或預測。

**若出現「無法連線至 POS 伺服器」**：請檢查 `.env`、網路、VPN 與資料庫權限。

## 二之一、Firebase 同步（POS_Sync_Tool）

**目的**：從 POS 資料庫抓取商品主檔，經加工後上傳至 Firebase Firestore，供前端或內部系統使用。

**前提**：專案根目錄需有 **`serviceAccountKey.json`**（Firebase 服務帳號金鑰），且 `.env` 可連線至 POS 資料庫。

**指令**：

```bash
python POS_Sync_Tool.py
```

**執行後會發生的事**：

1. 連線至 POS 資料庫，抓取商品庫存主檔（同 `get_stock_master_data()`）。
2. **上傳至 Firestore `products` collection**：僅含基本資訊（ProductCode, Barcode, Name, CurrStock, RetailPrice, Category, Supplier），供對外或一般查詢使用。
3. **加工並上傳至 Firestore `replenishment` collection**：經 `replenishment_service.prepare()` 解析 Note（第一次入貨量、文字描述），並寫入 LastInCost, AvgCost, InboundLocation, 第一次入貨量, Note描述 等補貨建議用欄位；此 collection 建議僅供內部補貨決策使用。

**Firestore 兩大 collection 對照**：

| Collection      | 用途說明           | 主要欄位（示意） |
|-----------------|--------------------|------------------|
| `products`      | 商品基本資訊       | ProductCode, Name, CurrStock, RetailPrice, Category, Supplier |
| `replenishment` | 補貨建議用敏感資訊 | ProductCode, LastInCost, AvgCost, InboundLocation, 第一次入貨量, Note描述 |

**若出現「無法連線至 POS 伺服器」**：同 **二、資料同步**，請檢查 `.env` 與網路。**若 Firebase 上傳失敗**：請確認 `serviceAccountKey.json` 路徑與權限。

## 三、進階：ABC/XYZ 分類與策略標籤（選用）

**目的**：依利潤與銷量波動做商品分級（A/B/C、X/Y/Z、新品），並產出策略建議（如「自動補貨」「AI 季節性預測」），可上傳至 Firebase 供內部系統使用。

**前提**：已執行過 **二、資料同步**，且 `DetailGoodsStockToday.csv`、`vw_GoodsDailySales_cache.parquet` 存在。

**指令**：

```bash
python business_insights.py
```

**輸出**：

- **`data/insights/abc_xyz_analysis.csv`**：各商品的 ABC/XYZ 分類、策略標籤等。
- 若專案有設定 Firebase，分類結果會上傳至 Firestore。

---

## 四、進階：銷售預測與建議進貨量（選用）

**目的**：預測下個月需求量，並計算「建議採購量」（目標庫存 − 當前庫存），產出補貨計畫表。

**前提**：已執行過 **二、資料同步**，且存在 `DetailGoodsStockToday.csv`、`vw_GoodsDailySales_cache.parquet`。

**流程（兩步驟）**：

| 步驟 | 說明 |
|------|------|
| 1 | 先執行 **二、資料同步**（`python pos_system_v2.py`），確保庫存與銷售資料為最新。 |
| 2 | 再執行補貨預測腳本。 |

**指令**：

```bash
python replenishment_forecasting.py
```

**輸出**：

- **`data/insights/final_inventory_plan.csv`**  
  內含欄位：商品代碼、名稱、類別、ABC_XYZ、基礎預測需求量、類別季節加成、最終需求量、安全庫存、目標庫存、當前庫存、**建議採購量（Suggested_Order）** 等。  
  可依此表作為採購或補貨參考。

**若出現「找不到輸入檔案」**：代表尚未執行或未成功執行 **二、資料同步**，請先完成 `pos_system_v2.py` 再跑本腳本。

**備註**：專案另提供 GPU 版 NeuralProphet 腳本（`replenishment_forecasting_GPU NeuralProphet.py`），邏輯相同，可依環境選用。

## 五、輸出檔案總覽

| 檔案路徑 | 產生程式 | 用途簡述 |
|----------|----------|----------|
| `data/processed/table_structure.txt` | pos_system_v2.py | 資料表結構說明（除錯/了解用） |
| `data/processed/DetailGoodsStockToday.csv` | pos_system_v2.py | 今日商品庫存主檔（分析與預測的輸入） |
| `data/processed/vw_GoodsDailySales_cache.parquet` | pos_system_v2.py | 銷售歷史快取（分析與預測的輸入） |

| `data/insights/final_inventory_plan.csv` | replenishment_forecasting.py | 補貨計畫（含建議採購量） |

## 六、建議執行順序（總整理）

1. **第一次**：完成 **一、前置準備** → 執行 **二、資料同步** → 視需求執行 **三** 或 **四**。
2. **之後每次要預測或報表前**：先執行 **二、資料同步**，再執行 **三** 或 **四**。
3. 僅要「庫存與銷售資料更新」時，只執行 **二** 即可。
4. **若要更新 Firebase（products + replenishment）**：執行 **二之一、Firebase 同步**（`python POS_Sync_Tool.py`）；建議在 **二、資料同步** 之後執行，以確保上傳資料為最新。
