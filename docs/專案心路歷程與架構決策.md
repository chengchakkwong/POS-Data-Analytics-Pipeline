# 專案心路歷程與架構決策

本文件記錄「自動化採購預測系統」從痛點到最終方案的演進脈絡與關鍵決策，供交接、回顧與新成員理解「為什麼這樣設計」時參考。

---

## 一、起點：資料庫效能與架構的權衡

### 痛點

Python 每兩週上傳「銷售預測」，每天上傳「最新庫存」，Vercel 網頁端該如何最有效率地計算並顯示「庫存缺口」？

### 思考過程

- 原本猶豫：要讓 Web 讀取兩份文件再計算，還是由 Python 先算好再上傳？
- 需同時考慮 Firebase (NoSQL) 的計費（讀取次數）與效能。

### 決策：客戶端計算 (Client-side Calculation)

- **做法**：將預測量與庫存量寫入**同一個 Document** 中。
- **效果**：Web 端只需 **1 次讀取**，就能即時在前端秒速算出差額（庫存缺口）。
- **理由**：結合 Firestore 的計費與效能特性，用「少讀取、一次拿齊」換取前端計算的即時性與成本可控。

---

## 二、進階：權限隔離與 NoSQL 資料冗餘設計

### 痛點

系統中有些詳細資料（如進貨成本、供應商資料）只能給主管看，但「預測缺口」需要讓一般員工看。Firebase 的權限是以「文件 (Document)」為單位，**無法隱藏單一欄位**。

### 思考過程

- 若只存一份 document，無法做到「同一份資料、不同人看不同欄位」。
- 必須從「資料如何寫入、如何分存」著手。

### 決策：資料冗餘 (Data Redundancy) / Fan-out 寫入

- **做法**：Python 在上傳時，透過 **Batch Write** 同步更新兩個獨立的 Collection：
  - **products（公開）**：僅含庫存與預測等非敏感資訊，開放員工讀取。
  - **replenishment（私密）**：包含成本、供應商等敏感資訊，透過 Security Rules 限制僅管理員可讀取。
- **術語**：從資料工程角度，同一來源寫入多處即「Fan-out 寫入」／寫時冗餘；本專案依「用途＋權限」拆成兩 collection，兼顧安全與讀取效能。
- **細節**：欄位對照、Security Rules 設定建議見 [Firestore_雙Collection設計與安全說明.md](Firestore_雙Collection設計與安全說明.md)。

## 三、深水區：新品冷啟動 (Cold Start) 與歷史數據提煉

### 痛點

對於**沒有足夠歷史數據**的商品，系統該如何給出**第一次入貨的建議量**？若已有零星歷史，直接算平均數或中位數不可行——歷史資料充滿雜訊（樣品單、打錯、湊免運等假性規律）。

### 思考過程

- 引入供應商的物理限制：**最低起訂量 (Min)** 與 **下單倍數 (Multiple)** 作為防呆底線。
- 對於已有零星歷史的商品，希望從歷史中「提煉」出 Min / Multiple。

### 技術嘗試與限制

- 曾考慮用 Python 純演算法：以「第一筆入貨紀錄」搭配 **最大公因數 (GCD)**、**差值法** 自動推算 Min 與 Multiple。
- **結論**：商業世界邊緣情況多（供應商中途改包裝、單位混淆如箱 vs 個等），純演算法無法 100% 準確，需保留人工把關。

## 四、最終解法：人機協作機制 (Human-in-the-Loop)

### 定位轉變

從「全自動取代人力」轉為 **「AI 輔助的人工決策系統」**：系統負責常規、人力負責例外與商業判斷。

### 運作機制

| 階段 | 說明 |
|------|------|
| **系統猜測 (System Guess)** | Python 腳本自動清理歷史數據、排除極端雜訊，並用公因數邏輯推算一組 `guessed_min` 與 `guessed_multiple`。 |
| **UI 呈現與人工把關** | Vercel 網頁端撈出「未確認」商品，向使用者展示系統猜測結果。 |
| **一鍵決策** | **贊同**：若猜得對，點擊【👍 贊同】，資料建檔。**修改**：若遇極端狀況（如供應商剛換包裝），點擊【✏️ 修改】，輸入正確數字後存檔。 |

### 商業價值

- **效率**：系統處理約 80% 常規商品，省去大量建檔時間。
- **風險控管**：人類商業直覺防禦約 20% 會導致採購災難的例外狀況。

## 五、補貨預測腳本：記憶體與算力策略（`replenishment_forecasting_GPU NeuralProphet.py`）

### 痛點

補貨預測腳本需對數千個 SKU 各自跑時間序列預測（NeuralProphet／Prophet）。若每個 worker 都收到「整張銷售大表」、且啟用 GPU，會出現：多進程序列化複製大表導致 RAM 爆炸；GPU 在「短序列、高並發」情境下 overhead 大，且 PyTorch 多進程易死鎖。

### 決策

| 項目 | 做法 | 理由 |
|------|------|------|
| **記憶體** | 進入多進程前先按 `GoodsID` 切碎成 `sales_by_sku` 字典，每個任務只傳該 SKU 的小 DataFrame | 避免 joblib/loky 對大表重複 pickle／複製，峰值記憶體從「大表 × worker 數」降為「一份切碎資料 + 小表 × worker 數」 |
| **算力** | 強制 NeuralProphet 使用 `accelerator="cpu"`，worker 數依 `cpu_count()-2` 動態設定 | 單一 SKU 僅十幾筆月資料，GPU 啟動與搬運成本大於運算；CPU 多核對「短平快」千次迴圈更穩定、總體更快 |
| **演算法** | X 類（需求最穩定）與 Y 類一併納入 AI 預測分支（`xyz in ['X','Y']`） | X 類季節性訊號最乾淨，交給時間序列模型可提升預測品質，避免誤用簡化規則 |

### 取捨與後續

- 主行程會常駐一份按 SKU 切碎的資料，換取 worker 不再持有大表副本。  
- 若業務上「缺月 = 0 銷量」，可考慮在 X/Y 分支對月序列補齊 0 再訓練，讓模型正確學淡季；若缺月為資料缺漏，則維持現狀。  
- 詳細改動紀錄見 `DevLog.md` 2025-03-17 條目。
