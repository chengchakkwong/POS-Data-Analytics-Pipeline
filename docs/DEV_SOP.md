# 開發 SOP（標準作業流程）

統一「開發前 → 開發中 → 開發後」的節奏，方便對照與交接。

---

## 一、開發前（Before）

### 1. 先看什麼

| 項目 | 說明 | 對照文件 |
|------|------|----------|
| 待辦與目標 | 這次要做的功能／修復／優化是什麼 | `DevLog.md` 文末「下次要做的優化」、專案 issue／任務 |
| 既有邏輯 | 要改的模組怎麼運作、依賴誰 | 相關 `.py`、`README`、`docs/使用說明.md` |
| 約定與風格 | 專案怎麼命名、怎麼 commit、日誌怎麼寫 | `docs/GIT_COMMIT_STYLE_私用.md`、`DevLog.md` 格式說明 |
| 環境與依賴 | 需要哪些套件、DB／API 設定 | `requirements.txt`、`.env` 或設定檔 |

### 2. 開發前檢查清單（可打勾）

- [ ] 已確認本次目標（功能／bug／重構）
- [ ] 已看過要改的程式與相關文件
- [ ] 已知 commit 風格（可 @ `GIT_COMMIT_STYLE_私用.md` 請 AI 產出）
- [ ] 本地可跑、依賴與設定無誤

### 3. 待做（Todo）怎麼記

- **短期**：寫在 `DevLog.md` 當日或文末「下次要做的優化」
- **較大項**：可開 issue 或列在 `docs/專案追蹤模板_老闆版.md`（若有使用）
- 開發前先掃一輪，避免重複做或漏做

---

## 二、開發中（想法 → 實踐 → Bug／改善）

### 流程概覽

```
想法／需求
    ↓
實踐（實作、小步提交）
    ↓
發現 Bug 或可改善點
    ↓
修復／重構／優化 → 再實踐
    ↓
穩定後進入「開發後」
```

### 1. 想法

- 把「要做什麼、為什麼」寫清楚（可先在 DevLog 記一筆）
- 若會動到多個檔案，先想好介面與依賴，再寫程式

### 2. 實踐

- 小步實作、可測就測，避免一次改太大
- 有雷或取捨記得隨手記在 DevLog（踩了什麼雷、為什麼這樣改）

### 3. 發現 Bug／想改善

- **Bug**：能重現就記步驟或範例；修完在 DevLog 寫一筆「修了什麼、原因」
- **改善**：效能、可讀性、架構調整 → 歸在「refactor」或「fix」，並在 DevLog 留紀錄

### 4. 開發中可對照

- 不確定 commit 怎麼寫 → @ `docs/GIT_COMMIT_STYLE_私用.md` + 「寫 git」
- 不確定日誌寫什麼 → 看 `DevLog.md` 格式說明：今天做了什麼、為什麼這樣改、踩了什麼雷、下次要做的優化

---

## 三、開發後（After）

### 1. 要寫／要更新的文件

| 類型 | 做什麼 | 對照 |
|------|--------|------|
| **日誌** | 在 `DevLog.md` 補一則：今天做了什麼、改了什麼、為什麼、踩雷與後續優化 | 見 DevLog 格式說明 |
| **Commit** | 用專案約定的類型與多 `-m` 條列寫清楚 | `docs/GIT_COMMIT_STYLE_私用.md` |
| **使用說明** | 若行為或流程有變（例如新腳本、新步驟），更新 `docs/使用說明.md` 或 README | 依專案習慣 |
| **API／規格** | 若有對外介面或規格變更，更新對應文件或註解 | 依專案習慣 |

### 2. 開發後對照表（打勾用）

- [ ] 已在 `DevLog.md` 記錄本次變更與心得
- [ ] 已依 `GIT_COMMIT_STYLE_私用.md` 完成 commit（可請 AI 產出指令）
- [ ] 若有介面／流程變更，已更新使用說明或相關文件
- [ ] 待做清單已更新（完成項劃掉、新項補上）

## 五、一句話版

- **開發前**：看目標、看程式、看風格與待做。
- **開發中**：想法 → 實踐 → 遇 bug/改善就修、隨手記 DevLog。
- **開發後**：寫 DevLog、按風格 commit、有變更就更新說明與待做。
